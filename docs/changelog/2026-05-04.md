## 1. 관측성 — `routes/_db-errors.js` 헬퍼 + 7 라우트 + ws.js 일괄 적용

**커밋**: `c6ad2c2`, `bfb5b38`, `e49a354`, `13d41b9`, `115bc47`

### 배경
필테#2 보고 'DB 에러' — 추측 트리거(호스트 짧은 시간 다수 방 생성·삭제 반복) 는 `/close` 가 status='closed' 만 set 하므로 실제 발생 경로 아님. 또한 frontend 에 'DB 에러' 문자열 grep 0건 — 사용자가 본 메시지는 다른 일반 토스트일 가능성. advisor 권고: 추측 픽스 대신 관측성 강화 후 다음 보고에서 SQLSTATE 식별.

### 헬퍼 (`routes/_db-errors.js`)
- `captureDbError(label, err, ctx)` — 로깅 + Sentry 캡처만, 응답은 호출 측 결정
- `logDbError(res, label, err, ctx)` — captureDbError + 표준 응답 `{error:'db error', code:<SQLSTATE>}`
- 두 헬퍼 모두 `err.code · err.detail · err.constraint · err.routine · err.table · err.message` 구조화 console.error + `Sentry.captureException(err, { extra: { route: label, ...pgFields, ctx } })`

### 적용 (49 HTTP catch + 4 WS catch)
- `rooms.js` — 5 게스트 진입 엔드포인트 try-catch wrap (GET `/rooms/:code`·`/preview`·`/me`·`/members` + POST `/join` pre-query) + 기존 4 catch 일원화
- `survey.js` — 7 catch logDbError (응답 표준화)
- `auth.js` — 5 catch captureDbError (기존 `{error:'INTERNAL_ERROR'}` 응답 보존, Sentry 캡처만 추가)
- `ai.js` — 2 catch captureDbError (기존 한국어/영어 사용자 메시지 응답 보존)
- `admin.js` — 12 외부 catch logDbError (PG 내부 leak 차단, A3 로 별도 분리되었음 — §3 참고). JSON parse 2 catch 는 DB 무관이라 제외
- `panel.js` — 14 외부 catch logDbError + 1 bulk-delete 내부 루프 captureDbError (failed[] 응답 포맷은 admin 진단용 err.message 보존)
- `admin-auth.js` — callback (redirect) + `/admin/me` (`'db_error'` 응답) 2 catch captureDbError
- `ws.js` — 4 catch captureDbError: initial sync (DB query 실패), waiting_too_long handler, state catch-up, host grace auto-close. 기존 connection-level Sentry.captureException 은 유지

## 2. A1 · 모바일 재입장 후기 페이지 튕김 픽스

**커밋**: `6af02d1`

### 원인
`public/guest.html` WS onclose 핸들러 (`fdddce9` 2026-04-18 '방종료설문보장') 의 정규식 `/closed/i.test(e.reason || '')` 이 4005 미도달 케이스 방어용으로 추가됐으나, 브라우저/OS 기본 close reason ("Connection closed before received" 등) 이 false-positive 매칭. 모바일 슬립 → visibilitychange → forceReconnect → onclose 발생 순간 후기 페이지로 튕김.

### 픽스 (Option B-lite, advisor 권고)
- `e.code === 4005` 명시: 즉시 redirect 유지
- `reason` 매칭만의 경우: REST `/api/rooms/:code` 로 `room.status` 재검증 (AbortSignal 2s timeout)
- timeout/네트워크 실패 시 transient으로 판단 → 기존 reconnect 로직에 fall-through (자동 재시도)
- AbortSignal.timeout iOS 16+ / Chrome 103+ — 타겟 디바이스 충분

### 검증
브라우저 onclose 동작이라 CLI smoke 불가. **사용자 실기기 검증**:
1. DevTools console → `STATE.ws.close(1006, 'Connection closed before received')` → redirect 없이 reconnect 메시지
2. `STATE.ws.close(4005, 'Room closed')` → /survey redirect 정상
3. 실기기 5분+ 백그라운드 → 포그라운드 → redirect 없음

## 3. A2 + A3 · 호스트 투표 집계 catch-up + admin 응답 정규화

**커밋**: `d13bb8c` (A2), `2b2a4de` (A3)

### A2 — 호스트 투표 catch-up
- **원인**: 호스트 WS 가 좀비 상태일 때 게스트 투표 변경 broadcast 미도달. 재접속 시 동기화 로직 부재 → state.currentVotes stale.
- **픽스**:
  - `forceReconnectHost` 에 `state.currentVotes = {}` pre-clear (race 제거)
  - `ws.onopen` 에 `catchUpVoteCounts()` 호출 — `/api/rooms/:code/explore-result` 로 모든 멤버 votes_json 재집계 → state.currentVotes 풀카운트 재구성 → 현재 보이는 explore 문항은 즉시 renderVoteBars
  - AbortSignal 3s timeout, 실패 시 다음 vote_result 메시지로 자동 복구

### A3 — admin.js 응답 정규화
- 기존 `{error: err.message}` 12 catch — PG 제약명·테이블명·SQL fragment 노출 가능 → 표준 `{error:'db error', code:<SQLSTATE>}` 로 일원화 (logDbError)
- host.html 기존 fallback toast (`err.message || '변경 실패'` 등) 그대로 노출 — `err.error === 'MIN_MEMBERS'` 같은 코드 매칭은 영향 없음
- captureDbError + custom 응답 (A 루트) → logDbError (B 루트) 로 변경
